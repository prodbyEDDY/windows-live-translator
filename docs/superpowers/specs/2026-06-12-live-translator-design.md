# Live Translator — спецификация

**Дата:** 2026-06-12
**Платформа:** Windows 10 2004+ / Windows 11
**Стек:** Tauri 2.x (Rust backend) + React/TypeScript/HeroUI frontend
**Статус:** утверждено пользователем 2026-06-12

---

## 1. Обзор продукта

Десктоп-приложение для Windows — живой двунаправленный переводчик звонков и голосовых сообщений. Пользователь выбирает свой язык и язык собеседника; приложение в реальном времени:

- переводит речь собеседника (звук из Zoom/WhatsApp/Telegram/браузера) на язык пользователя и проигрывает в его наушники;
- переводит речь пользователя (микрофон) на язык собеседника и подаёт в звонок через виртуальный микрофон (VB-CABLE);
- транскрибирует и переводит брошенные в окно голосовые сообщения (WhatsApp и др.);
- записывает голосовое в приложении, переводит его и отдаёт переведённый аудиофайл, который можно мышкой перетащить в мессенджер.

Один вендор — Google AI Studio. Пользователь вставляет свой API-ключ, может сменить его в любой момент.

### 1.1 Используемые модели

| Задача | Модель | Канал |
|---|---|---|
| Live-перевод звонка (каждое направление — отдельная сессия) | `gemini-3.5-live-translate-preview` | Live API, WebSocket `BidiGenerateContent` |
| Транскрибация + перевод голосовых (текст) | `gemini-3.5-flash` | REST `generateContent` |
| Синтез переведённого голосового | `gemini-3.1-flash-tts-preview` | REST `generateContent` |

Ориентир стоимости live-перевода: ~$0.023/мин на направление (~$0.046/мин при активных обоих направлениях).

### 1.2 Не-цели (v1)

- Не macOS/Linux (только Windows).
- Не «нативные» голосовые WhatsApp — перетащенный файл придёт как аудиовложение (формат нативных голосовых закрыт).
- Не собственный виртуальный аудиодрайвер — используем VB-CABLE.
- Не диаризация нескольких собеседников — входящий поток переводится как единый источник.
- Не мобильные/веб-версии.

## 2. Архитектура

Принятое решение (вариант A): **весь аудиотракт и WebSocket-сессии живут в Rust-бэкенде**; фронтенд — только UI, получает события (транскрипты, уровни, статусы) и шлёт команды через Tauri IPC. Аудиоданные через IPC не передаются (кроме воспроизведения карточек голосовых, которые читаются как файлы через asset protocol).

```
┌─────────────────────────── Tauri App ───────────────────────────┐
│  ┌──────────── Frontend (WebView2) ────────────┐                │
│  │ React + TS + HeroUI + Tailwind + i18next    │                │
│  │ экраны: Live / Voice / History / Settings   │                │
│  └────────────▲──────────────────┬─────────────┘                │
│        events │ (transcripts,    │ commands (start/stop,        │
│               │  levels, status) │  settings, voice msg ops)    │
│  ┌────────────┴──────────────────▼─────────────┐                │
│  │              Rust Core (tokio)              │                │
│  │  audio::capture   — WASAPI mic + process    │                │
│  │                     loopback                │                │
│  │  audio::playback  — наушники + CABLE Input  │                │
│  │  audio::resample  — rubato 48k↔16k/24k      │                │
│  │  audio::ducking   — ISimpleAudioVolume      │                │
│  │  gemini::live     — 2× WS-сессии, lifecycle │                │
│  │  gemini::rest     — generateContent (STT/   │                │
│  │                     перевод/TTS)            │                │
│  │  voice::codec     — Opus/OGG encode/decode  │                │
│  │  store            — settings.json, SQLite,  │                │
│  │                     keyring (API key)       │                │
│  └─────────────────────────────────────────────┘                │
└──────────────────────────────────────────────────────────────────┘
```

### 2.1 Модули Rust (crate `src-tauri`)

| Модуль | Ответственность | Ключевые зависимости |
|---|---|---|
| `audio::devices` | Перечисление устройств ввода/вывода, отслеживание подключения/отключения (IMMNotificationClient), перечисление приложений с активными аудиосессиями (IAudioSessionManager2), детект VB-CABLE | `windows` crate |
| `audio::capture` | Захват микрофона (WASAPI shared), захват звука процесса (`ActivateAudioInterfaceAsync` + `AUDIOCLIENT_ACTIVATION_TYPE_PROCESS_LOOPBACK`, include-tree по PID), фолбэк — loopback всего устройства с exclude-tree собственного процесса | `windows` crate |
| `audio::playback` | Два независимых выхода: наушники пользователя и CABLE Input; джиттер-буфер на приём от Gemini; микшер (опция «подмешать оригинал» в CABLE) | `windows` crate |
| `audio::resample` | 48 кГц устройства → 16 кГц моно PCM16 (вход модели); 24 кГц (выход модели) → частота устройства | `rubato` |
| `audio::ducking` | Плавное снижение громкости аудиосессии целевого приложения до настроенного уровня пока играет перевод, восстановление после тишины (рамп ~150 мс, восстановление после 400 мс пустого буфера) | `windows` crate |
| `gemini::live` | Менеджер WS-сессии: setup, стриминг чанков по 100 мс, приём аудио+транскриптов, session resumption, обработка GoAway, реконнект с буферизацией исходящего аудио на время разрыва | `tokio-tungstenite`, `serde_json` |
| `gemini::rest` | `generateContent`: транскрибация+перевод аудиофайлов (inline base64 ≤ 20 МБ, иначе Files API), TTS; проверка ключа (`models.list`) | `reqwest` |
| `voice::codec` | Декод входных форматов (ogg/opus, mp3, m4a/aac, wav, flac) для длительности/волны; запись микрофона → Opus в OGG-контейнере 16 кГц/48 кГц | `symphonia`, `opus`, `ogg` |
| `store` | `settings.json` в app-data; история в SQLite (`rusqlite`); API-ключ в Windows Credential Manager (`keyring`), миграция/очистка | `rusqlite`, `keyring`, `serde` |
| `wizard` | Логика мастера: статус ключа/кабеля/устройств, запуск инсталлятора VB-CABLE (скачивание официального zip, запуск от админа), loopback-тест «скажи и услышь» | — |

### 2.2 Tauri IPC — команды и события

**Команды (frontend → Rust):**

```
settings_get() -> Settings
settings_set(patch: Partial<Settings>)
api_key_set(key: string) -> KeyStatus        // сохраняет в keyring, валидирует через models.list
api_key_status() -> KeyStatus                // present | missing | invalid
devices_list() -> { inputs: Device[], outputs: Device[], cable_present: bool }
audio_apps_list() -> AppSession[]            // приложения с активным звуком: pid, имя, иконка
live_start(cfg: LiveConfig)                  // языки, источник (pid | устройство), флаги
live_stop()
voice_import(path: string) -> VoiceMsgId     // drop файла
voice_record_start() / voice_record_stop() -> VoiceMsgId
voice_translate(id, targetLang) -> VoiceMsgResult   // транскрипт + перевод + путь к TTS .ogg
voice_drag_out(id)                           // нативный drag файла (tauri-plugin-drag)
history_list(filter) / history_clear()
wizard_state() -> WizardState
wizard_install_cable()                       // скачать и запустить инсталлятор VB-CABLE
wizard_loopback_test_start/stop()
```

**События (Rust → frontend):**

```
live:state        { phase: idle|connecting|running|reconnecting|error, perSession: {...} }
live:transcript   { direction: in|out, kind: original|translated, text, isFinal }
live:levels       { micDb, appDb, outDb }          // ~10 Гц
live:cost         { seconds, estimatedUsd }
voice:progress    { id, stage: decoding|transcribing|translating|synthesizing|done|error }
devices:changed   { ... }                          // hotplug
error             { code, message, recoverable }
```

## 3. Аудиоподсистема — детали

### 3.1 Исходящий тракт (вы → собеседник)

1. Захват выбранного микрофона: WASAPI shared mode, нативный формат устройства (обычно 48 кГц float32).
2. Конверсия → моно PCM16 16 кГц (rubato, sinc-качество).
3. Нарезка чанками по 100 мс (3200 байт) → `realtimeInput` в WS-сессию **OUT** (`targetLanguageCode` = язык собеседника).
4. Приём `serverContent` аудио 24 кГц → джиттер-буфер (целевая глубина 150–250 мс) → ресемплинг под CABLE Input → воспроизведение в устройство **CABLE Input**.
5. В Zoom/WhatsApp пользователь выбирает микрофоном **CABLE Output** (мастер настройки показывает инструкцию со скриншотами).
6. Опция «подмешивать оригинал» (выкл. по умолч.): микрофонный сигнал с гейном −18..−6 дБ микшируется в CABLE под переводом — собеседник слышит интонацию оригинала.

### 3.2 Входящий тракт (собеседник → вы)

1. Источник по выбору пользователя:
   - **Приложение (по умолчанию):** Process Loopback по PID выбранного приложения (include-tree — захватываются и дочерние процессы, важно для браузеров).
   - **Всё устройство (фолбэк):** loopback дефолтного устройства вывода с `PROCESS_LOOPBACK_MODE_EXCLUDE_TARGET_PROCESS_TREE` для собственного PID — наш переведённый звук не попадает обратно в модель.
2. → 16 кГц PCM16 → WS-сессия **IN** (`targetLanguageCode` = язык пользователя).
3. Приём перевода → джиттер-буфер → выбранное устройство вывода пользователя (наушники).

### 3.3 Ducking (заглушение оригинала)

- Динамический: как только в джиттер-буфере сессии IN появляется аудио — громкость аудиосессии целевого приложения (ISimpleAudioVolume) плавно (150 мс) опускается до `duckLevel` (слайдер 0–100%, по умолчанию 20%); после 400 мс пустого буфера — плавный возврат.
- Если источник «всё устройство» — дакается мастер-громкость всех чужих сессий, кроме нашей.
- При остановке перевода/краше приложения громкости **обязательно** восстанавливаются (RAII-guard в Rust).
- Отдельная настройка — `echoTargetLanguage` (Gemini API): если собеседник уже говорит на языке пользователя, модель либо «эхо-повторяет» речь (true), либо молчит (false, по умолчанию). Объясняется подсказкой в настройках.

### 3.4 Защита от петель и эха

- Захват по приложению не видит наш вывод по построению; фолбэк-захват исключает наш процесс.
- Акустическая петля (микрофон слышит колонки с переводом): мастер настройки и экран Live показывают предупреждение, если устройство вывода — не наушники (эвристика по имени/форм-фактору устройства); рекомендация наушников.
- Сессия OUT получает только микрофон, сессия IN — только звук приложения: потоки никогда не смешиваются на входе.

### 3.5 Устойчивость

- Отключение устройства (наушники/микрофон выдернули): пауза тракта, событие `devices:changed`, тост с предложением выбрать другое устройство; автовозобновление, если устройство вернулось.
- VB-CABLE пропал — сессия OUT останавливается с понятной ошибкой и ссылкой на мастер.
- Изменение дефолтного устройства Windows во время работы не влияет (держим конкретные device ID).

## 4. Интеграция с Gemini

### 4.1 Live-сессии

URL: `wss://generativelanguage.googleapis.com/ws/google.ai.generativelanguage.v1beta.GenerativeService.BidiGenerateContent?key=<API_KEY>`

Setup-сообщение (по одной сессии на направление):

```json
{
  "setup": {
    "model": "models/gemini-3.5-live-translate-preview",
    "generationConfig": {
      "responseModalities": ["AUDIO"],
      "inputAudioTranscription": {},
      "outputAudioTranscription": {},
      "translationConfig": {
        "targetLanguageCode": "<BCP-47, напр. ru | en | de>",
        "echoTargetLanguage": false
      }
    }
  }
}
```

- Аудио внутрь: `realtimeInput.audio = { data: <base64 PCM16 16k mono LE>, mimeType: "audio/pcm;rate=16000" }`, чанки 100 мс.
- Наружу: аудио 24 кГц PCM16 + транскрипты входа/выхода → события `live:transcript` (оригинал и перевод, маркировка по направлению).
- **Lifecycle:** включаем session resumption; по `GoAway`/разрыву — немедленный реконнект с resumption handle; исходящие чанки на время разрыва буферизуются (кольцевой буфер ≤ 10 с) и доотправляются. UI показывает фазу `reconnecting`, не падает.
- Языки: два дропдауна «Мой язык» / «Язык собеседника» из списка 70+ поддерживаемых (константа в коде с автонимами: «Русский», «English», «Deutsch»…), кнопка свапа.
- Сессии стартуют/останавливаются только парой (кнопка Start/Stop), но независимо переживают реконнекты.

### 4.2 Голосовые: транскрибация и перевод

`POST /v1beta/models/gemini-3.5-flash:generateContent` с inline аудио (≤ 20 МБ; больше — Files API: upload → fileUri). Один запрос, структурированный промпт: «верни JSON: { sourceLang, transcript, translation }», перевод на язык пользователя (для drop) или язык собеседника (для записанных). Парсинг — толерантный (модель без structured-output режима в этом сценарии — валидируем и ретраим 1 раз при битом JSON).

Поддерживаемые входные форматы: `.ogg/.opus` (WhatsApp), `.mp3`, `.m4a/.aac`, `.wav`, `.flac` (декод длительности/пиков — `symphonia`; в Gemini уходит исходный файл как есть, mime по расширению).

### 4.3 Голосовые: синтез перевода

`POST /v1beta/models/gemini-3.1-flash-tts-preview:generateContent`, текст перевода → аудио; выбор голоса в настройках (список prebuilt-голосов, по умолчанию нейтральный). Результат конвертируется в **OGG/Opus 48 кГц моно ~32 кбит/с** (как у WhatsApp) и сохраняется в `app-data/voice/<id>.ogg`.

### 4.4 API-ключ

- Хранение: Windows Credential Manager через `keyring` (DPAPI). В settings.json ключ не пишется никогда.
- Ввод/смена: поле в настройках (маскированное) + первый экран мастера. Кнопка «Проверить» → `GET /v1beta/models` → valid/invalid с причиной (401/403/quota).
- Все ошибки квоты/биллинга (429, RESOURCE_EXHAUSTED) показываются человеческим текстом со ссылкой на AI Studio.

## 5. Голосовые сообщения — UX-поток

### 5.1 Drop (входящее голосовое)

1. Файл перетащен в окно (Tauri file-drop) → карточка появляется мгновенно: имя, длительность, волна, стадия `transcribing`.
2. По готовности: транскрипт оригинала + перевод на язык пользователя, кнопки «копировать текст», «прослушать оригинал».
3. Ошибки (неподдерживаемый формат, > лимита, сеть) — в самой карточке с кнопкой retry.

### 5.2 Запись и отправка (исходящее голосовое)

1. Кнопка записи (push-to-talk или toggle) → запись с выбранного микрофона, таймер, волна, лимит 5 мин.
2. Стоп → стадии: транскрипт → перевод (на язык собеседника) → TTS → карточка результата: переведённый текст + плеер переведённого аудио.
3. На карточке — **drag-хэндл**: нативный drag-out файла `.ogg` (`tauri-plugin-drag`) — бросаем прямо в окно WhatsApp/Telegram. Также «Сохранить как…» и «Показать в папке».
4. Подпись-дисклеймер: «придёт как аудиофайл, не как голосовая заметка».

## 6. UI

Светлая тема, HeroUI (React), Tailwind. Минимализм: одна основная колонка, навигация — сайдбар с 4 пунктами. i18n: русский (по умолчанию) + английский, переключение в настройках, `i18next`.

### 6.1 Экран Live (главный)

- Шапка: [Мой язык ▾] ⇄ [Язык собеседника ▾], выбор источника звука собеседника (дропдаун приложений с иконками + пункт «Весь звук системы»), большая кнопка **Start/Stop**.
- Лента субтитров: хронологическая, пузыри двух цветов (вы/собеседник), в каждом — оригинал (приглушённый) и перевод (основной), частичные транскрипты дорисовываются на лету.
- Статус-бар: индикаторы двух сессий (зелёный/жёлтый-reconnecting/красный), уровни микрофона и приложения, таймер звонка и оценка стоимости ($).
- Предупреждающий баннер, если: нет VB-CABLE (с кнопкой в мастер), вывод не наушники, ключ невалиден.

### 6.2 Экран Голосовые

Дроп-зона на весь экран при перетаскивании; кнопка записи; вертикальный список карточек (см. §5).

### 6.3 Экран История

Список звонков (дата, языки, длительность, полный транскрипт по клику) и голосовых (карточки повторно открываемы, файлы переживают перезапуск). Поиск по тексту. Кнопка «Очистить историю» (подтверждение).

### 6.4 Экран Настройки

- API-ключ: маскированное поле, «Проверить», «Сменить».
- Аудио: микрофон, устройство вывода, статус VB-CABLE (+ кнопка переустановки/мастера).
- Перевод: echoTargetLanguage (свитч с пояснением), ducking on/off + слайдер уровня, «подмешивать оригинал» on/off + слайдер.
- Голосовые: голос TTS, формат имени файла.
- Приложение: язык UI, очистка истории, версия.

### 6.5 Мастер первого запуска (4 шага)

1. **Ключ:** поле + ссылка «получить в AI Studio» + проверка.
2. **VB-CABLE:** автодетект; если нет — кнопка «Скачать и установить» (официальный дистрибутив, запуск инсталлятора с UAC), после установки — перепроверка.
3. **Устройства:** микрофон + наушники, предупреждение про колонки.
4. **Тест:** loopback-проба — «скажите фразу» → перевод проигрывается в наушники; галочка «всё работает». Инструкция: «в Zoom выберите микрофон CABLE Output» со скриншотом.

Мастер можно перезапустить из настроек.

## 7. Хранение данных

| Что | Где |
|---|---|
| API-ключ | Windows Credential Manager (keyring) |
| Настройки | `%APPDATA%/live-translator/settings.json` |
| История (звонки, сообщения, транскрипты) | `%APPDATA%/live-translator/history.sqlite` |
| Аудиофайлы голосовых (оригиналы и переводы) | `%APPDATA%/live-translator/voice/` |

Очистка истории удаляет SQLite-записи и файлы. Аудио звонков **не записывается** на диск никогда (только транскрипты).

## 8. Ошибки и edge cases

| Ситуация | Поведение |
|---|---|
| Ключ отсутствует/невалиден | Блокирующий баннер + переход к вводу ключа; Start недоступен |
| 429 / квота исчерпана | Стоп сессий, понятное сообщение, ссылка на биллинг AI Studio |
| Разрыв сети / GoAway | `reconnecting` ≤ 30 с с буферизацией; дальше — стоп с ошибкой |
| VB-CABLE не установлен | Сессия OUT недоступна, баннер с мастером; сессия IN работает (односторонний режим допускается) |
| Целевое приложение закрылось | Захват останавливается, тост, автопредложение выбрать другой источник |
| Устройство выдернуто | Пауза + автовосстановление при возврате |
| Бросили не-аудио файл | Карточка-ошибка «формат не поддерживается» |
| Файл > 20 МБ | Автоматически через Files API (прозрачно для пользователя) |
| Краш приложения при активном ducking | RAII-восстановление громкостей; на старте — sweep и сброс заниженных нами сессий |
| Два экземпляра приложения | single-instance plugin, фокус существующего окна |

## 9. Тестирование

- **Rust unit:** ресемплер (частоты/длины), нарезка чанков, парсинг сообщений Live API (фикстуры JSON), кольцевой буфер, JSON-парсер ответов транскрибации, RAII ducking-guard.
- **Rust integration (ручные/полуавтоматические, помечены `#[ignore]`):** WASAPI capture/loopback на реальном железе, реальный WS-коннект с тестовым ключом (smoke: setup → 2 с тишины → закрытие).
- **Frontend:** vitest + testing-library на сторы и ключевые компоненты (лента субтитров с частичными транскриптами, карточка голосового по стадиям).
- **E2E-сценарий (ручной чек-лист в репо):** полный звонок Zoom ↔ телефон, drop голосового WhatsApp, запись → drag-out в WhatsApp.
- Mock-сервер Live API (локальный WS) для тестов реконнекта/GoAway без расхода квоты.

## 10. Этапы разработки

**Этап 1 — MVP live-перевода:**
скелет Tauri + UI-каркас (Live, Settings) → ключ + keyring + проверка → перечисление устройств/приложений → захват микрофона → сессия OUT → воспроизведение в CABLE → process loopback → сессия IN → наушники → ducking → реконнекты → мастер настройки.

**Этап 2 — голосовые сообщения:**
drop + транскрибация/перевод → запись в приложении → TTS → OGG/Opus → drag-out → история (SQLite).

**Этап 3 — полировка:**
оценка стоимости в UI, подмешивание оригинала, локальный VAD (пауза стрима в тишине — экономия до ~50%), single-instance, NSIS-инсталлятор, автообновления (tauri-updater), EN-локализация добивается до 100%.
